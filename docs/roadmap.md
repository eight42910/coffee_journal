# Coffee Journal - 完全学習ロードマップ

**目標**: このプロジェクト完了時点で、35 歳未経験からエンジニアとして引っ張りだこになる JS 基礎力を獲得する

---

## なぜこのプロジェクトで「引っ張りだこ」になれるのか

### 一般的な未経験エンジニアとの差別化ポイント

#### ❌ 一般的な未経験者

- チュートリアルをなぞっただけ
- コピペで動かしただけ
- エラーが出たら詰まる
- 「なぜ動くか」を説明できない
- テストを書いたことがない
- フレームワークに依存しすぎ

#### ✅ あなた（このプロジェクト完了後）

- **基礎から実装できる**（DOM 操作、配列メソッド、非同期処理）
- **設計思想を理解している**（状態管理、純関数、不変性）
- **テストを書ける**（Vitest、カバレッジ 80%以上）
- **デバッグができる**（コンソール、エラーハンドリング）
- **パフォーマンスを意識している**（デバウンス、メモ化）
- **コードレビューに耐えられる**（命名、構造、保守性）

### 35 歳でも引っ張りだこになる理由

1. **基礎力が圧倒的**

   - 若手より深い理解で、技術負債を作らない
   - フレームワークが変わっても対応できる

2. **問題解決能力**

   - 「動かない理由」を論理的に特定できる
   - ドキュメントを読んで自己解決できる

3. **コミュニケーション能力**

   - コードレビューで「なぜこう書いたか」を説明できる
   - 技術選定の根拠を述べられる

4. **実装速度**
   - 基礎が固まっているので、新機能の実装が速い
   - 車輪の再発明をせず、適切にライブラリを選択できる

### 2. 面接での技術説明例

#### 質問: 「この機能はどう実装しましたか？」

**❌ ダメな回答**:
「検索機能は filter を使って実装しました」

**✅ 良い回答**:
「検索機能は、入力値を小文字に統一して `filter` で部分一致検索しています。
パフォーマンスのため、デバウンスで 300ms 待ってから実行します。
また、検索結果を 0 件の場合と、データ自体が 0 件の場合で表示を分けています。
テストでは、空文字、大文字小文字、日本語、特殊文字のケースをカバーしています」

#### 質問: 「なぜこの技術選択をしましたか？」

**❌ ダメな回答**:
「みんなが使っているから」

**✅ 良い回答**:
「localStorage を選んだ理由は、バックエンドなしで永続化でき、学習コストが低いためです。
将来的には IndexedDB やサーバー連携も視野に入れていますが、
まずはシンプルな実装で基礎を固めることを優先しました。
実務では要件に応じて、Cookie、SessionStorage、IndexedDB を使い分けます」

## よくある挫折ポイントと対策

### 挫折ポイント 1: Phase 2 の localStorage

**症状**: データが保存されない、読み込めない

**対策**:

1. ブラウザの開発者ツールで確認
2. `console.log` でデータの流れを追跡
3. try-catch でエラーをキャッチ
4. 手順書のテストコードを 1 つずつ実行

### 挫折ポイント 2: Phase 3 の配列メソッド

**症状**: filter/sort/reduce が理解できない

**対策**:

1. 小さなサンプルで 1 つずつ試す
2. MDN のドキュメントを読む
3. console.log で途中結果を確認
4. 紙に書いて処理の流れを追う

### 挫折ポイント 3: Phase 4 のテスト

**症状**: テストが通らない、書き方がわからない

**対策**:

1. まずは 1 つのテストだけ書く
2. 正常系から始める（異常系は後回し）
3. 手順書のサンプルをコピペして動かす
4. 徐々にテストケースを追加

### 挫折ポイント 4: Phase 6 のリファクタリング

**症状**: ファイル分割で import/export が混乱

**対策**:

1. 1 つのファイルから始める
2. 動作確認しながら少しずつ分割
3. エラーメッセージをよく読む
4. VS Code の自動補完を活用
